01 System Architecture

Distributed microservices architecture with Spring Boot as the API gateway and core services, Django handling admin analytics and document processing, and a shared PostgreSQL layer per bounded domain context.

Spring Boot

Auth Service

JWT issuance, RBAC enforcement, password encryption (BCrypt). Owns the users table and roles.

Spring Boot

Claims Service

Full lifecycle management: submit → review → approve/reject. Workflow state machine + audit logs.

Spring Boot

Policy Service

Policy CRUD, coverage validation, policy-to-claim linkage, coverage limit enforcement.

Django

Document Service

File upload (PDF, JPEG, PNG), metadata storage, S3 integration, virus scanning hooks.

Django

Admin Dashboard

Analytics, reporting, agent workflows, bulk approval, charts (Chart.js), KPI tracking.

Gateway

API Gateway

Single entry point. Routes, rate-limits, JWT validation, cross-cutting CORS, request logging.

Inter-Service Communication

Sync (REST)

All client-facing operations. Gateway validates JWT then proxies to the correct service on internal ports. Spring services expose on :808x, Django on :900x.

Async (Event-Driven)

Claims status changes publish events consumed by the Notification service. Document upload triggers validation events. Redis pub/sub or a lightweight queue (RabbitMQ optional).

Shared Contracts

Each service validates incoming requests via its own schema. No shared DB across service boundaries. Django calls Spring APIs via internal HTTP when needed.

02 Database Design

Per-service schema separation. All tables use UUID primary keys for distribution safety. Timestamps on every entity. Soft deletes via deleted_at.

users

Column Type Constraints Notes
PK id UUID PRIMARY KEY, DEFAULT gen_random_uuid() User identity
email VARCHAR(255) UNIQUE, NOT NULL Login credential
password_hash VARCHAR(255) NOT NULL BCrypt hash
full_name VARCHAR(200) NOT NULL
phone VARCHAR(20) UNIQUE
role ENUM NOT NULL, DEFAULT 'USER' USER | ADMIN | AGENT
is_active BOOLEAN DEFAULT true
created_at TIMESTAMPTZ NOT NULL, DEFAULT NOW()
updated_at TIMESTAMPTZ NOT NULL Auto-update trigger
deleted_at TIMESTAMPTZ NULLABLE Soft delete

policies

Column Type Constraints Notes
PK id UUID PRIMARY KEY
FK user_id UUID REFERENCES users(id) Policy holder
policy_number VARCHAR(50) UNIQUE, NOT NULL Human-readable ID
plan_name VARCHAR(200) NOT NULL e.g. Gold Health Plus
coverage_amount DECIMAL(12,2) NOT NULL, CHECK > 0 Max claimable amount
deductible DECIMAL(12,2) NOT NULL, DEFAULT 0
premium DECIMAL(10,2) NOT NULL Monthly premium
start_date DATE NOT NULL
end_date DATE NOT NULL
status ENUM NOT NULL, DEFAULT 'ACTIVE' ACTIVE | EXPIRED | CANCELLED

claims

Column Type Constraints Notes
PK id UUID PRIMARY KEY
FK user_id UUID REFERENCES users(id) Claimant
FK policy_id UUID REFERENCES policies(id) Linked policy
FK assigned_agent_id UUID REFERENCES users(id), NULLABLE Reviewing agent
claim_number VARCHAR(50) UNIQUE, NOT NULL Auto-generated e.g. CLM-20240001
amount_claimed DECIMAL(12,2) NOT NULL, CHECK > 0
amount_approved DECIMAL(12,2) NULLABLE Set on approval
description TEXT NOT NULL Nature of claim
incident_date DATE NOT NULL When medical event occurred
status ENUM NOT NULL, DEFAULT 'PENDING' PENDING | UNDER_REVIEW | APPROVED | REJECTED | CANCELLED
rejection_reason TEXT NULLABLE
submitted_at TIMESTAMPTZ NOT NULL, DEFAULT NOW()
reviewed_at TIMESTAMPTZ NULLABLE
resolved_at TIMESTAMPTZ NULLABLE

documents

Column Type Constraints Notes
PK id UUID PRIMARY KEY
FK claim_id UUID REFERENCES claims(id)
FK uploaded_by UUID REFERENCES users(id)
file_name VARCHAR(500) NOT NULL Original filename
storage_key VARCHAR(1000) NOT NULL S3 object key
file_size BIGINT NOT NULL Bytes
mime_type VARCHAR(100) NOT NULL application/pdf etc.
doc_type ENUM NOT NULL MEDICAL_BILL | PRESCRIPTION | DISCHARGE | OTHER
uploaded_at TIMESTAMPTZ NOT NULL, DEFAULT NOW()

claim_audit_logs

Column Type Constraints
PK id UUID PRIMARY KEY
FK claim_id UUID REFERENCES claims(id)
FK actor_id UUID REFERENCES users(id)
previous_status VARCHAR(50) NOT NULL
new_status VARCHAR(50) NOT NULL
notes TEXT NULLABLE
created_at TIMESTAMPTZ NOT NULL, DEFAULT NOW()

notifications

Column Type Constraints
PK id UUID PRIMARY KEY
FK user_id UUID REFERENCES users(id)
title VARCHAR(300) NOT NULL
message TEXT NOT NULL
is_read BOOLEAN DEFAULT false
type ENUM CLAIM_UPDATE | POLICY_EXPIRY | GENERAL
created_at TIMESTAMPTZ NOT NULL

Relationships: users 1→M policies · users 1→M claims · policies 1→M claims · claims 1→M documents · claims 1→M audit_logs · users 1→M notifications

Request Body

Validation Rules
  • email: valid format, unique
  • password: min 8 chars, 1 uppercase, 1 special
  • phone: E.164 format
  • role: only USER allowed on self-register

  "userId":  "uuid-abc-123",

  "message": "Registration successful"
}

  • 409 — email already exists

Django-served analytics page with Chart.js: status donut, monthly bar chart, claims table with bulk approve/reject.

Agent View

Claim queue filtered to assigned claims. Quick status update buttons, notes field, document preview panel.

Axios Interceptor Setup

// api/axiosConfig.js
const api = axios.create({ baseURL: process.env.REACT_APP_API_URL });

api.interceptors.request.use(config => {
  const token = useAuthStore.getState().accessToken;
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});

api.interceptors.response.use(
  res => res,
  async err => {
    if (err.response?.status === 401) {
      await refreshAccessToken();
      return api(err.config);
    }
    return Promise.reject(err);
  }
);

06 Authentication & Security

JWT-based stateless authentication with refresh token rotation, BCrypt password hashing, and method-level RBAC via Spring Security annotations.

JWT Token Structure

// Header
{ "alg": "HS512", "typ": "JWT" }

// Payload (access token, 1hr TTL)
{
  "sub":      "uuid-user-123",
  "email":   "amit@example.com",
  "role":    "USER",
  "iat":     1710925200,
  "exp":     1710928800
}

// Refresh token: 7-day TTL, stored server-side
// Refresh token rotation: old token invalidated on use

Spring Security Config

// Endpoint-level role control
@PreAuthorize("hasRole('ADMIN')")
public ResponseEntity updateClaimStatus(...) {}

// SecurityFilterChain
.requestMatchers("/api/auth/**").permitAll()
.requestMatchers("/api/admin/**").hasRole("ADMIN")
.requestMatchers("/api/claims/**")
  .hasAnyRole("USER","AGENT","ADMIN")
.anyRequest().authenticated()

Security Best Practices Checklist

Area Implementation Priority
Passwords BCryptPasswordEncoder(strength=12) Critical
JWT Storage Access token in memory, refresh in httpOnly cookie Critical
HTTPS Enforce TLS in production via nginx/load balancer Critical
CORS Whitelist only frontend origin, no wildcard High
Rate Limiting Login: 5 attempts / 15 min per IP High
Input Validation Bean Validation (JSR-380) + Django serializers High
SQL Injection JPA/Hibernate parameterized queries, Django ORM High
File Upload MIME type check, file size limit, rename on save Medium
Audit Logging Log all state changes with actor + timestamp Medium
Secrets Environment variables / Docker secrets, never in code Critical

Naming Convention

Git Workflow Steps

1
Feature branch off develop — git checkout -b feature/your-task develop
2
Commit often — conventional commits: feat:, fix:, refactor:, test:, docs:
3
Pull Request to develop — require 1 teammate review before merge. PR description links the task.
4
Merge to main on Day 7 — squash merge for clean history. Tag release: git tag v1.0.0-mvp
5
Protected branches — main and develop require PR + passing CI (run tests, lint)

AWS EC2/ECS or DigitalOcean App Platform. RDS PostgreSQL. ElastiCache Redis. S3 for documents. ALB as entry point with SSL termination.
